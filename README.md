Сервис бронирования переговорок по спецификации api.yaml.

Стек: Go, PostgreSQL, Docker Compose. Точка входа cmd/server, логика в internal/app. Схема БД встроена и накатывается при старте.

Сделано: REST API с chi и pgx; авторизация через POST /dummyLogin (JWT с user_id и role, фиксированные UUID для admin и user). Администратор создаёт переговорки и одноразовые расписания, смотрит все брони с пагинацией. Пользователь смотрит переговорки, доступные слоты на дату, свои будущие брони, создаёт и отменяет свои брони. Слоты по 30 минут из расписания, стабильные UUID в базе, ограничения из задания (одна активная бронь на слот, UTC, идемпотентная отмена и т.д.). GET / и GET /_info отвечают 200. Юнит- и интеграционные тесты, покрытие выше 40%.

Не входило в объём: эндпоинты register и login; опциональная ссылка на конференцию — заглушка URL без внешнего сервиса.

Запуск из корня репозитория: docker compose up --build. Сервис слушает порт 8080.

Принятые решения. Слоты не строятся на весь календарь заранее: при запросе списка на дату они считаются из расписания комнаты, UUID детерминирован (SHA-1 от room_id и начала интервала в UTC), строки пишутся в slots с ON CONFLICT DO NOTHING — идентификаторы лежат в БД, а лишние даты не накапливаются. Уникальность пары (room_id, start_ts) исключает пересечения слотов одной переговорки. Схема накатывается при старте одним встроенным SQL. Серверная логика в одном пакете internal/app.
